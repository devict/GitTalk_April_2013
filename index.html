Why VCS?

  • Reversibility
  • Concurrency
  • Annotation
  • Peer Pressure

LOCO vs MOM

  • Lock On Checkout (RCS, SCCS)
  • Merge On Modify
    • Merge Before Commit (CVS, Subversion)
    • Commit Before Merge (Bazaar, Mercurial, Git)

Centralized vs Decentralized

  • Centralized VCS have a single server against which developers interact
  • Decentralized VCS give each user a complete copy of the repository

Where Did Git Come From?

This Guy

Linux Kernel Maintenance

  • Kernel was originally managed by email patches
  • Started using BitKeeper in 2002
  • In 2005, BitKeeper revoked its free status
  • Work on Git started on April 3, 2005
  • Project officially announced April 6, 2005
  • Self-hosting on April 7, 2005

Core Values

  • Non-linear development
  • Decentralized
  • Can handle large scale projects (like the kernel)
  • Cryptographically secure (use of SHA1 hashing)

  • Allows you to review the state of the repository.
  • Files aren't tracked by git automatically. They must be staged
  • Once tracked, git will notice changes, but you must specify which changes to commit
  • You can commit portions of changes, so a file may exist in multiple stages.

Review Your Changes

  • You can review your changes with the git diff command
  • Displays changes in the unified diff format
  • Without any flags, it displays the changes to the unstaged files
  • Add the --cached flag to show the changes to any staged files

Common Flags

  • --cached : Show changes to staged files
  • --name-only : Show the names of the changed files
  • --unified=n : Show n lines of context in the diff
  • --ignore-all-space : Ignore whitespace changes
  • --word-diff : Show diffs by words instead of by lines

Diffing Ranges

  • You can review the diffs of a range of commits with git diff SHA1..SHA1
  • You can also select relative to tags or relative to HEAD
$ git diff 1ece66b..d2949c7
$ git diff v1.8.2.0..v1.8.2.1
$ git diff HEAD~10..HEAD

Show

  • You can review the diff of any commit by using git show SHA1
  • You can review the diffs of a range of commits with git show SHA1..SHA1
  • You can also select relative to tags or relative to HEAD
$ git show d2949c7
$ git show 1ece66b..d2949c7
$ git show v1.8.2.0..v1.8.2.1
$ git show HEAD~10..HEAD

Show vs Diff

  • The syntax for git show and git diff are very similiar.
  • git show shows the diffs on a per commit basis over the specified range
  • git diff shows the combined difference over the specified range

Stage this hunk [y,n,q,a,d,/,j,J,g,s,e,?]?

Lots of Options

y - stage this hunk
n - do not stage this hunk
q - quit; do not stage this hunk nor any of the remaining ones
a - stage this hunk and all later hunks in the file
d - do not stage this hunk nor any of the later hunks in the file
g - select a hunk to go to
/ - search for a hunk matching the given regex
j - leave this hunk undecided, see next undecided hunk
J - leave this hunk undecided, see next hunk
k - leave this hunk undecided, see previous undecided hunk
K - leave this hunk undecided, see previous hunk
s - split the current hunk into smaller hunks
e - manually edit the current hunk
? - print help

Most Are Self-Explanatory

What about this option 'e'?

# ---
# To remove '-' lines, make them ' ' lines (context).
# To remove '+' lines, delete them.
# Lines starting with # will be removed.
#
# If the patch applies cleanly, the edited hunk will immediately be
# marked for staging. If it does not apply cleanly, you will be given
# an opportunity to edit again. If all lines of the hunk are removed,
# then the edit is aborted and the hunk is left unchanged.

Ignore Files

  • There may be some files you don't want to ever commit
  • You could just never git add them, but that's error prone, especially with git add .
  • You could just git reset them, but that's just as error prone
  • Git provides a better way, through git ignore files

Git Ignores

  • Each line in the git ignore file specifies a file or pattern to exclude
  • Three options for git ignores
    • System wide git ignore 
      $ git config --global core.excludesfile '~/.gitignore'
    • Two version of repo specific ignores
      • .gitignore which is used to exclude files specific to the project
      • .git/info/exclude which is used to exclude files specified to working environment

Committing Con't

  • Conventions for commits
    • First line is a summary, stopping at 80 characters
    • Blank Line
    • Lines 3+ contain a description of the changes / issue
  • Various commit options
$ git commit --message='Commit message' # Single line commit message
$ git commit --all # Automatically add all tracked files and open commit buffer
$ git commit --amend # Modify the prior commit message
$ git commit --author='Author <email@example.com>' # Override the author for the commit

Delete A File

  • Deleting a file is pretty straightforward
  • You can use the OS command to remove the file 
    # On branch master
#
# Changes not staged for commit:
#   (use "git add/rm ..." to update what will be committed)
#   (use "git checkout -- ..." to discard changes in working directory)
#
#       deleted:    commit.h
#
no changes added to commit (use "git add" and/or "git commit -a")
  • As you can see, git still have to be told you want to stage the deletion

git rm

  • git rm to stage a deleted file
  • Alternatively, it can replace the use of the OS command, doing all operations at once.
  • # On branch master
#
# Changes to be committed:
#   (use "git reset HEAD ..." to unstage)
#
#       deleted:    commit.h
#

Rename a File

  • Renaming or moving a file is a little different
  • You can use the OS command to move the file
  • Git still has to be told about this change. 
    # On branch master
# Your branch is ahead of 'origin/master' by 314 commits.
#
# Changes not staged for commit:
#   (use "git add/rm ..." to update what will be committed)
#   (use "git checkout -- ..." to discard changes in working directory)
#
#       deleted:    commit.h
#
# Untracked files:
#   (use "git add ..." to include in what will be committed)
#
#       commit_new.h
no changes added to commit (use "git add" and/or "git commit -a")
  • This is tedious and hurts our ability to look at the history of the file

A Better Way to Move

  • You can use the git mv file new_path
  • This handles the delete / add steps from above
  • Also has the added benefit that git log now knows about the move

Git Revert

  • git reset modifies history, which is a dangerous thing
  • git revert undoes a commit without destroying history
  • It generates a new commit which undoes the prior commit
$ git log --oneline
0e356b9 Committing main.cpp
0edeab9 Variable rename
$ git revert 0e356b9
$ git log --oneline
e60a298 Revert "Committing main.cpp"
0e356b9 Committing main.cpp
0edeab9 Variable rename

Concurrent Development Branches

  • One of the places where git shines is branches
  • Branching allows for experimentation and concurrent development without polluting the main branch
  • Every repo gets a branch for free, master
  • The name is immaterial; you could rename it to trunk if you're more comfortable with that terminology

Which Branches Are Available

$ git branch
  maint
* master
  next
  • The * represents the branch you are currently on.
  • To switch to a branch, you use git checkout branch

Creating branches

  • Create a new branch from the current branch with 
    git branch new_branch
  • Create a new branch from a specific commit with 
    git branch new_branch SHA
  • Using a similar syntax, you can create and checkout the new branch with 
    git checkout -b new_branch

Deleting Branches

  • First switch to another branch
  • git branch -d branch_to_delete
  • If the branch hasn't been fully merged, you may receive an error.
  • If you're sure you want to delete it, replace the -d with -D as git reminds you
error: The branch 'next' is not fully merged.
If you are sure you want to delete it, run 'git branch -D next'.

A Couple More Commands

  • If you need to rename the branch, you can do 
    git branch -m old_branch new_branch
  • If you want to find out which branches contain a certain commit, you can 
    git branch --contains SHA

Merging Branches

  • Once you have finished work on a feature branch, you'll want to bring it back into the main developmet line.
  • This is known as merging, and is, appropriately, accomplished with the git merge command.

Automatic Merges

  • Git is quite intelligent about merging branches
  • Whenever possible, git will automatically apply the appropriate changesets
